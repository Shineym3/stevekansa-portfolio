Inventory Management: From "Ugh" to "Oh Wow!"

Home Depot needed to retire a 30-year-old ordering system. I inherited a struggling project, challenged what MVP meant, and helped ship something users actually loved.

RoleUX Designer
CompanyHome Depot
TypeEnterprise · Internal Tools
Outcome98% Data Accuracy · 90% Dev Time Reduction

Situation

Home Depot wanted to sunset an ordering system that had been in use since 1990. The new system needed to provide real-time order tracking and handle larger data sets. The legacy system had a hard limit on line items and would silently error out — orders would fail to place and items would disappear into a black hole with no notification to the user.

I inherited this project midstream. Another designer had already started the work using a legacy design library called Pallet. The designs were functional but uninspiring, and the development process was slow and stressful — Pallet required constant modification from the engineering team, adding friction at every sprint.

Task

The initiative aimed to: enable day-one no-training functionality for ordering assets, allow for faster processing times on orders, add real-time visibility into order statuses, and challenge what MVP meant based on user feedback rather than the product roadmap.

Research

User Interviews

I began with a review of the previous designer's research, then conducted in-person interviews with inventory admins. This was a small and specialized group: only five people used this system. Despite the small sample, the signal was strong — all five users reported virtually the same pain points, giving us high confidence in our findings. One turned out to be a super-user; the other four used it in limited capacity. That distinction shaped how I thought about different use cases.

Personas

Two personas were established from the interviews. Observational studies were conducted alongside interviews — watching users work in the system revealed insights that structured questions alone wouldn't have surfaced. Time-on-task was measured for each user group to understand how the system's inefficiencies affected each work stream differently.

Journey Mapping

Journey maps were created for four user types: General Tools, Merchandising, Vehicles, and Large Equipment. The mapping revealed that while all user groups reported similar pain points, the frequency with which they experienced those pains varied significantly by role. This informed how we prioritized which fixes would have the broadest impact.

Design

Picking Up the Pieces

The designs I inherited were built on Home Depot's legacy design system, Pallet. Developers had to constantly modify Pallet components, creating technical debt and slowing delivery. I saw an opportunity.

Wireframes

Rough sketches were used to quickly validate needed functionality with users. Given the small user group, this step took about a week. Sketches allowed rapid changes to be communicated with the product team without creating false expectations from polished mockups.

UI Design

I made a deliberate choice to move away from Pallet entirely and adopt the ANT design system. Development time was reduced by up to 90%. It also freed up half of our design team's capacity, which was reallocated to a new initiative — effectively cutting production costs in half. Mockups were delivered to developers with embedded instructions and direct links to ANT components.

What We Learned the Hard Way

The first iteration did not succeed. Near-zero user adoption.

The initial version lacked critical functionality, so users continued using the old MMTR software. This failure taught the team an important lesson: we rescoped based on what users actually needed. The second iteration addressed those gaps and led to a complete overhaul — and genuine user delight.

Outcomes & Impact

98%Data accuracy — up from frequent silent errors that caused orders to disappear
300msAverage processing time per order
90%Reduction in development time by adopting ANT design system over legacy Pallet
2Iterations — the first surfaced critical gaps; the second delivered a complete overhaul users loved

What We Learned

Inheriting a project comes with unique challenges. Starting over on the design system — while counterintuitive — was the right call. It unlocked development speed, reduced team stress, and freed up capacity for other work. Failing early on the first iteration was painful but instructive. The second iteration succeeded precisely because we took the first failure seriously.
